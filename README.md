# JTBD Analyzer

> 基于 JTBD（Jobs To Be Done）框架，对访谈文本、访谈笔记和用户反馈材料进行结构化分析，提取 Job Statements、情境卡片与设计建议。

---

## 这是什么

`jtbd-analyzer` 是一个用于分析定性研究材料的 skill，适合处理访谈逐字稿、访谈笔记、用户反馈、研究记录等原始文本。

它的目标不是做一份泛泛的“访谈总结”，而是用 JTBD 框架把材料拆成更可操作的研究输出：

- **Job Statements**
- **情境卡片**
- **竞争方案分析**
- **四力分析（推力 / 拉力 / 焦虑 / 惯性）**
- **洞察与设计建议**

---

## 适用场景

这个 skill 适合以下任务：

- 分析用户访谈
- 提取 JTBD 洞察
- 从原始访谈中提炼用户动机
- 识别用户在特定情境下“想完成什么 job”
- 将研究材料转成结构化分析报告

如果用户没有明确说 JTBD，但给了你访谈文本、笔记或用户反馈，并希望理解用户需求、动机或行为模式，这个 skill 也适用。

---

## 输出内容

默认会输出一份结构化 JTBD 分析报告，包括：

### 1）执行摘要
- 材料数量
- 核心 Job 数量
- 关键发现

### 2）Job Statements 列表
按三层 job 拆分：
- Functional Job
- Emotional Job
- Social Job

### 3）情境卡片
每个核心 job cluster 会整理成情境卡，包含：
- Job Statement
- 触发情境
- 竞争方案
- 四力分析
- 代表性原话

### 4）洞察与设计建议
- 核心洞察
- 对产品 / 设计 / 策略的含义
- 可执行的建议方向

### 5）附录编码表
保留原始 job 信号，方便研究员复核。

## 工作方式

### 1）先收集材料
必须提供：
- 一份或多份访谈文本 / 访谈笔记

可选但推荐提供：
- 研究背景
- 受访者信息
- 分析重点

### 2）先逐份提取，再跨份聚合
如果有多份材料，skill 会先做单份提取，再做跨案例聚合，识别重复出现的 job pattern 和差异性 job。

### 3）严格区分证据与推论
它会尽量把“用户原话”“研究观察”“研究解释”分开，避免把研究员脑补直接写成结论。

---

## 质量标准

这个 skill 强调几件事：

- **Job Statement 必须可操作**，而不是模糊标签
- **竞争方案视角要足够宽**，不只盯着同类产品
- **每个核心洞察都应有原文依据**
- **设计建议必须可落地**，避免空泛表达

同时会尽量避免这些常见错误：

- 把功能诉求直接当成 job
- 把 emotional job 写成人格标签
- 忽略替代方案和切换逻辑
